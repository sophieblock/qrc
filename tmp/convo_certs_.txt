user: When am I required to update trust chain (new issuing and root CA) certificates in my trust store?
assistant: Short answer
- Update your trust store whenever a new Boeing CA (root or issuing/intermediate) is deployed and you need your systems to trust certificates issued by that new CA.
- Keep existing CA certs in place until all certificates that chain to them have expired or are no longer in use.

What that means in practice
- Immediately: when Boeing publishes a new trust chain (for example during the KeyFactor migration), check your application and server trust stores and add the new root and intermediate CA certificates so your clients and servers will trust newly-issued certs.
- Don’t remove the old CA certs until every system that relies on them is transitioned — removing them too soon can break TLS, client auth, code-signature validation, etc.
- Audit cadence: verify CA trust lists at least monthly (or more often during migrations/rollouts) and after any PKI change announcement.

Boeing-specific guidance & links
- New BAS trust chain example (download and install these if required):
  - Boeing BAS Root CA SHA256 G3: http://crl.keyfactorpkigeo.com/Boeing%20BAS%20Root%20CA%20SHA256%20G3.crt
  - Boeing BAS Issuing CA SHA256 G4: http://crl.keyfactorpkigeo.com/Boeing%20BAS%20Issuing%20CA%20SHA256%20G4.crt
  - (Device issuing for isolated systems) Boeing Windows Device Issuing CA G3: http://crl.keyfactorpkigeo.com/Boeing%20Windows%20Device%20Issuing%20CA%20G3.crt
- Ensure these PKI endpoints are reachable from your environment:
  - KeyFactor Command (RA): https://boeing.keyfactorpki.com
  - CDP/AIA: http://crl.keyfactorpki.com
  - OCSP: http://boe.keyfactorpki.com/ocsp
- CMI/KeyFactor user guidance and user guide: https://boeing.sharepoint.us/sites/IAMPKI and Boeing Keyfactor CMI User Guide (link available from the PKI Information Page).

Quick checklist to update trust chain
1. Identify affected systems/apps & owners.
2. Download new root/intermediate certs from the AIA or links above.
3. Install them to the appropriate trust stores (OS, JVM, appliance, application).
4. Verify CDP/AIA/OCSP access and test certificate validation (browser, curl, app).
5. Keep old CA certs until all relevant end-entity certs are migrated/expired.
6. Document the change and confirm monitoring/alerting for revocation/validation.

- For process questions outside PKI/CMI, use AskBoeing: https://boeing-chat.moveworksgov.com/

user: How do I request a CMI team to be added?
assistant: Use the BES Enterprise Services Catalog to request a new CMI team. Steps and required info:

1. Open the PKI catalog request:
   - https://besweb.web.boeing.com/ISCatalogServices/Form?tid=161
   - Select: PKI → PKI Consultation/General Request

2. Fill the form:
   - Subject: CMI – Create Team <teamname>
   - Description: include the requested team name and list team members and their BEMSID(s). Minimum 2 members required.
   - Provide an Email Distribution List (DL) that will receive certificate notifications for the team (individual emails are not accepted). If you don’t have a DL, request one in MARS: https://mars.web.boeing.com/mars/dlhome?6
   - Any other required justification/details per the form

3. Submit. PKI will create the team in the pki.notification.<teamname> format and assign the Owner Role (OwnerRoleName) used by KeyFactor.

After creation
- Manage membership via IDAP Teams: https://idap.web.boeing.com/Teams/
- You’ll receive instructions when the team and OwnerRole are ready.

assistant: Problem summary
- pip failed to reach your corporate Artifactory because Python/openssl could not verify the server TLS chain:
  [SSL: CERTIFICATE_VERIFY_FAILED] certificate verify failed: self-signed certificate in certificate chain
- That means the HTTPS server (sres.web.boeing.com) presents a cert chain that includes a CA not trusted by your Python/openssl trust store.

Quick remediation options (ranked)
1) Preferred — add the Boeing CA(s) to a trust bundle Python/openssl uses
   - Fetch the Boeing root and intermediate CA(s) that sign the Artifactory cert (example BAS links are provided in Boeing PKI docs).
   - Create a PEM file that concatenates the root and any intermediates: boeing-ca-chain.pem
   - Tell pip/requests/openssl to use that bundle:
     - export SSL_CERT_FILE=/path/to/boeing-ca-chain.pem
     - OR export REQUESTS_CA_BUNDLE=/path/to/boeing-ca-chain.pem
     - OR for pip only: pip3 install --cert /path/to/boeing-ca-chain.pem poetry
   - Test: python3 -c "import ssl; print(ssl.get_default_verify_paths())" and curl -v --cacert /path/to/boeing-ca-chain.pem https://sres.web.boeing.com/

2) macOS system trust (recommended for long-term if you use system Python)
   - Import the Boeing Root/Intermediate into Keychain Access and mark as Always Trust.
   - Restart terminal / your Python process and retry pip. This makes the OS trust the CA for apps using macOS trust store.

3) If using the certifi bundle (many Python installs use it)
   - Locate certifi cacert.pem:
     - python3 -c "import certifi; print(certifi.where())"
   - Append Boeing CA PEM(s) to that file (or replace with a bundle you control).
   - Retry pip.

4) Temporary (not recommended for production)
   - pip3 install --trusted-host sres.web.boeing.com poetry
   - Or set PIP_TRUSTED_HOST in pip config. This skips verification and is insecure — only use as a quick test.
