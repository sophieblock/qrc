Explanations / rationale and key details
- .condarc choices:
  - channel_alias sets the base alias that conda uses for named channels. Setting it to your artifactory is correct.
  - custom_channels maps named channel keys (like conda-forge) to a specific URL. This is how you ensure the named "Conda-Forge" channel resolves to sres rather than conda.anaconda.org.
  - default_channels: [] removes automatic fallback to repo.anaconda.com defaults. If you rely on defaults for certain packages, consider adding those explicitly but they will be external.
  - override_channels_enabled: False avoids ad-hoc channels from commands accidentally contacting upstream.
  - show_channel_urls: True helps you inspect which channel each package claims.
  - channel_priority: strict avoids mixing packages from multiple channels during resolves when possible.
- Why curl showed 403: 403 means “host reachable, but not allowed” — likely artifactory requires credentials, client certs, or specific headers. This is different from a routing issue. Your traceroute and nslookup showing Boeing network confirms routing is fine.
- Why you still see conda.anaconda.org in history: conda stores channel metadata on install and in revisions; if earlier installs used upstream or if some operations explicitly pointed to upstream, history will reflect that. Reinstalling or forcing re-downloads via the artifactory will change future metadata.

Troubleshooting tips if artifactory returns 403
- Attempt to open the artifactory URL in a browser while on VPN and see whether it prompts for login or shows content. If it requires SSO or credentials you may need to configure conda to use an API key or configure JFrog credentials in ~/.netrc or via conda's authentication integration.
- If the artifactory requires a client certificate, ensure your client cert is installed and that REQUESTS_CA_BUNDLE or SSL_CERT_FILE point to a CA bundle the artifactory trusts.
- Ask the artifactory/DevOps team whether anonymous read of repodata is allowed or whether reads require special headers or auth. They can tell you whether they intentionally block direct unauthenticated repodata access.
