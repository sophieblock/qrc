Quick summary
- Preferred method: Docker Desktop for Mac (Apple Silicon build).
- Alternatives (headless / CLI-only): Colima + Docker CLI, Rancher Desktop.
- Key arm64 considerations: use arm64 images or multi‑arch builds; emulation (qemu) for amd64 is supported but slower.

Prerequisites
- macOS on Apple Silicon (M1, M2, or later) — arm64 architecture.
- macOS version compatible with Docker Desktop (check Docker Desktop requirements; commonly macOS 12+ or 13+ depending on Docker version).
- Sufficient disk space (images and volumes can grow quickly) and memory for the VM/container runtime.

Install Docker Desktop (recommended)
1. Install Docker Desktop via Homebrew Cask:
   ```
   brew install --cask docker
   ```
2. Start Docker Desktop from Applications (you can also run `open /Applications/Docker.app`).
3. On first run, grant required permissions (e.g., virtualization, file access) when macOS prompts. Docker uses Apple’s virtualization.framework on Apple Silicon.
4. Confirm Docker is running by checking the whale icon in the menu bar and waiting for the “Docker Desktop is running” state.

Verify installation
- Check Docker version and platform:
  ```
  docker version
  docker info
  ```
- Run the test container:
  ```
  docker run --rm hello-world
  ```

Apple Silicon / arm64 specifics
- Prefer native arm64 images. Example official images often provide multi‑arch manifests that include arm64:
  `docker pull nginx:latest`
- If an image lacks arm64, Docker Desktop can use QEMU emulation to run amd64 images, but expect slower startup and runtime.
- Check the image architectures with:
  ```
  docker manifest inspect --verbose <image>:<tag>
  ```
- For builds that must support multiple architectures, use Buildx:
  ```
  docker buildx create --use --name mybuilder
  docker buildx build --platform linux/amd64,linux/arm64 -t myimage:multiarch --push .
  ```

Alternatives to Docker Desktop (headless / scriptable)
- Colima + Docker CLI: lightweight, good for CI/local headless workflows.
  ```
  brew install colima docker
  colima start --arch aarch64
  ```
  Then use `docker` as usual.
- Rancher Desktop: provides containerd/nerdctl or Docker-compatible CLI and Kubernetes.
- Use these if you prefer non‑GUI or need different runtime behavior.

Configuration and common settings
- Resources: Use Docker Desktop > Preferences > Resources to adjust CPU, memory, and disk image size.
- File sharing: Configure shared folders in Docker Desktop > Preferences > Resources > File Sharing. Note that bind‑mount performance to macOS host is slower than native Linux; use docker volumes for better performance in many cases.
- Daemon config:
  - Docker Desktop manages its daemon via UI. Adding `/etc/docker/daemon.json` on macOS may not affect Docker Desktop. Use the Docker Desktop UI for common settings (registry mirrors, insecure registries) or edit Docker Desktop’s advanced settings.
  - For Colima/Rancher Desktop you can configure their daemon settings via their own config files or CLI options.
- Logging rotation: configure in Docker Desktop Preferences or using `daemon.json` on non‑Desktop runtimes. Example `daemon.json`:
  ```json
  {
    "log-driver": "json-file",
    "log-opts": {
      "max-size": "10m",
      "max-file": "3"
    },
    "storage-driver": "overlay2"
  }
  ```

Image registry and credentials
- Use `docker login` as usual:
  ```
  docker login my-registry.example.com
  ```
- Docker Desktop integrates with macOS keychain for credential storage by default. Credential helpers are supported.
- For internal registries that are insecure (HTTP), configure via Docker Desktop > Preferences > Docker Engine (or via `daemon.json` for other runtimes). Mark insecure registries only if absolutely necessary.

Kubernetes (optional)
- Docker Desktop includes an optional lightweight Kubernetes cluster that can be enabled in Preferences > Kubernetes. This is useful for local testing but not for production workloads.
- For Kubernetes with Colima or Rancher Desktop, check those products’ Kubernetes support (Rancher Desktop includes k8s; Colima can enable k8s via --kubernetes).

Common commands and maintenance
- Check running containers:
  `docker ps -a`
- Inspect disk usage:
  `docker system df`
- Cleanup unused images, containers, networks, and build cache:
  ```
  docker system prune --all --volumes
  ```
- Remove a specific image:

Troubleshooting (Apple Silicon)
- Docker won’t start or VM fails:
  - Ensure virtualization permissions were granted in System Settings.
  - Check Docker Desktop logs via the app (Troubleshoot > Get Support) or:
    ```
    ~/Library/Containers/com.docker.docker/Data/log/vm
    ```
- File sharing or volume permission issues:
  - Ensure host path is included in Docker Desktop file sharing settings.
  - Remember user mapping: mounts map macOS user to container uid/gid; adjust ownership inside container as needed.
- Slow filesystem I/O:
  - Prefer Docker volumes over bind mounts for heavy IO workloads.
- Image or binary is amd64 only:
  - Use multi‑arch builds or explicit `--platform linux/amd64` with awareness of QEMU emulation overhead:
    `docker run --platform linux/amd64 --rm <image>`
- CLI not found after installing Docker Desktop:
  - Docker Desktop installs the CLI to PATH when the app runs. Start Docker Desktop, then open a new terminal session.
- If using Colima and Docker CLI reports connection issues:
  ```
  colima status
  colima start
  ```

Security & policies
- Avoid running containers as privileged unless necessary. Use least privilege for container capabilities.
- Follow Boeing internal policies for base images, vulnerability scanning, SBOMs, signing, and secrets management.
- Be careful with insecure registries and credential handling—use TLS and authenticated registries whenever possible.

Best practices for Apple Silicon
- Build multi‑arch images if images will be run on mixed arch fleets.
- Test images on arm64 hosts or CI runners that emulate arm64 to catch architecture-specific bugs early.
- Use `docker buildx` for reproducible multi‑arch builds and `--push` to upload multi‑arch manifests to a registry.
- For development workflows with heavy I/O, prefer volumes and keep hot code in volumes rather than bind mounts when performance matters.

What these variables do
- `CURL_CA_BUNDLE`: Path to a PEM file containing CA certificates used by `curl` when making TLS connections (overrides system bundle).
- `REQUESTS_CA_BUNDLE`: Path to a PEM file used by Python `requests` library to validate TLS (requests honors this env var).
- `PIP_CERT`: Path to a CA bundle for `pip` to use when connecting to HTTPS package indexes (or use `--cert` on the pip command line).
- `SRES_API_KEY_ENDPOINT`, `SRES_USERNAME`, `SRES_API_TOKEN` (Boeing-specific):
  - Likely internal variables used by Boeing SRES tooling to locate token endpoints and authenticate API requests. Treat as secrets and do not store them in plaintext in repos.

Where to put CA bundle files
- Place a PEM file with the internal CA(s) at a secure, system-appropriate path, e.g.:
  - `/usr/local/etc/ca-bundle.crt` (Homebrew prefix)
  - `~/certs/internal-ca.pem` (user-scoped)
- Ensure file permissions restrict access:
  - `chmod 600 /path/to/internal-ca.pem`
  - `chown $(whoami) /path/to/internal-ca.pem`

- For interactive shells and terminal apps, set and export in your shell profile (recommended: `~/.zprofile` for login shells, `~/.zshrc` for interactive; adapt for bash).

  `export SRES_USERNAME="sres-user"`

- Example block (add to `~/.zprofile`):
  ```bash
  export CURL_CA_BUNDLE="/opt/homebrew/etc/ssl/certs/internal-ca.pem"
  export REQUESTS_CA_BUNDLE="/opt/homebrew/etc/ssl/certs/internal-ca.pem"
  export PIP_CERT="/opt/homebrew/etc/ssl/certs/internal-ca.pem"
  ```

    ```xml
        <key>Label</key>
    ```
